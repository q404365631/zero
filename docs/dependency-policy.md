# Dependency And Supply Chain Policy

ZERO dependencies should stay boring, reviewable, and reversible. The public
runtime is safety software; convenience dependencies are not worth hidden
custody, release, or audit risk.

## Current Dependency Surface

- Python engine metadata lives in `engine/pyproject.toml`.
- Python live exchange support stays in the optional `live` dependency group.
- Rust CLI dependencies are locked in `cli/Cargo.lock`.
- GitHub Actions, Python, and Cargo updates are grouped by `.github/dependabot.yml`.
- Release SBOM and local provenance files are generated by
  `scripts/release_provenance.py`.

## Update Rules

1. Dependency PRs must pass `just ci`.
2. Runtime dependency additions need a short reason in the PR body.
3. Live-execution dependencies must stay optional unless the public paper
   runtime cannot function without them.
4. Major version bumps that touch signing, networking, serialization, storage,
   key handling, release, or CI require a safety-review issue or PR section.
5. Do not add generated lockfiles or vendored source trees outside the relevant
   package manager path.
6. Do not add package-registry tokens, API keys, exchange keys, or publish
   credentials to examples, docs, workflows, or local env templates.

## Release Requirements

Before tagging:

```bash
just ci
just registry-readiness
just release-rehearsal
```

The release bundle must include:

- `SHA256SUMS`
- `SBOM.spdx.json`
- `PROVENANCE.json`
- GitHub artifact attestations for release assets

`SBOM.spdx.json` is a dependency-free SPDX 2.3 document generated from Cargo,
Python, workspace, and release-asset metadata. `PROVENANCE.json` records the
source commit, dirty-state flag, release asset hashes, and policy assertions
such as paper-mode default and package-registry publication disabled.

## Vulnerability Response

For a suspected vulnerable dependency:

1. Classify severity using the incident levels in `docs/incident-runbooks.md`.
2. Identify whether the affected dependency is runtime, dev-only, CI-only, or
   live-optional.
3. If live execution, key handling, release, or serialization is affected,
   pause release promotion until the patch is verified.
4. Patch, pin, replace, or remove the dependency.
5. Regenerate lockfiles through the package manager, not manual edits.
6. Run `just ci`, `scripts/hardening_gate.sh`, and `just release-rehearsal`.
7. If a public release is affected, publish a patched release and mark the old
   release unsafe or keep it in draft.

## Acceptable Exceptions

An exception is acceptable only when all are true:

- the dependency is needed for a specific public runtime capability;
- the risk is documented in the PR;
- the dependency is covered by tests or smoke checks;
- the rollback path is clear;
- the dependency does not weaken paper-mode defaults or self-custodial live
  boundaries.
